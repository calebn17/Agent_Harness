# Agent Harness

A token-efficient, persistent context layer for AI coding agents. Lives in any project as a git submodule. Gives Claude Code, Cursor (Composer 2), and Codex a shared memory, codebase map, and convention rules — so every session starts informed, not cold.

---

## The Problem

AI agents forget everything between sessions. They re-explore code they've already seen, repeat corrected mistakes, and ignore project-specific rules. Each session wastes tokens rebuilding context that should already be known.

## The Solution

A lightweight harness that:
- Maintains a **persistent memory** of architecture decisions, patterns, and corrections
- Generates a **codebase map** so agents know where things are without exploring
- Injects a **~500 token bootstrap** at session start — only what each role needs
- Automates bookkeeping via scripts so agents spend tokens on actual work

---

## Installation

```bash
# Prerequisite: Python 3.9+, pyyaml
pip3 install pyyaml

# Add as git submodule to your project
git submodule add <this-repo-url> .agent-harness

# Run from your PROJECT ROOT directory
cd /path/to/your/project
.agent-harness/harness init --type nextjs   # or python, ios-swift, node
```

`harness init` (run from project root) will:
- Generate an initial codebase map
- Install git hooks for incremental map updates and automatic handoff generation
- Inject role-specific skills blocks into CLAUDE.md, .cursorrules, and AGENTS.md
- Create transient directories for reviews and handoffs

## Usage

### Session Workflow

Each agent role follows a **skill** — a markdown instruction file the agent reads at the start of every session.

**Planner session:**
```bash
Read: .agent-harness/skills/planner/planner-start.md
```

**Coder session:**
```bash
Read: .agent-harness/skills/coder/coder-start.md
```
This skill automatically:
- Checks for a pending handoff (auto-generated by post-commit hook after last commit)
- Ingests and deletes the handoff
- Checks for code reviews left in `.agent-harness/reviews/`
- Points to the implementation plan

**Reviewer session:**
```bash
Read: .agent-harness/skills/reviewer/reviewer-start.md
```
When review is complete:
```bash
Read: .agent-harness/skills/reviewer/leave-review.md
```
This creates a review doc that the coder picks up automatically.

### Additional Skills (invoked as needed)

Agents use these skills during their session:

- **Coder:** `query-memory.md`, `save-mistake.md`, `drill-map.md`, `pick-up-review.md`
- **Planner:** `create-plan.md`
- **Reviewer:** `leave-review.md`

### Memory & Context (on-demand)

Agents invoke from their skills:

```bash
harness memory query "authentication"         # retrieve relevant memories
harness memory save --category pattern "..."  # save something learned
harness mistakes <file>                       # check known pitfalls for a file
harness map drill "src/auth"                  # detailed module map
harness rules check                           # validate conventions
```

## Roles & Skills

| Role | Model | Skill Entry Point |
|---|---|---|
| **planner** | Claude Opus | `.agent-harness/skills/planner/planner-start.md` |
| **coder** | Sonnet / Cursor Composer 2 | `.agent-harness/skills/coder/coder-start.md` |
| **reviewer** | Codex | `.agent-harness/skills/reviewer/reviewer-start.md` |

Each role reads a **skill file** at session start. The skill:
- Loads filtered bootstrap context (map, conventions, mistakes, test results)
- Guides the agent through role-specific steps
- Directs them to other skills as needed
- Handles session bookkeeping automatically (handoffs, reviews, memory)

## Automatic Handoffs & Reviews

**Session handoff (automatic):**
- After every commit, the post-commit hook generates `.agent-harness/handoff/handoff.md`
- Contains: commit message, changed files, diff summary
- Next session's coder reads it via `coder-start.md`, then deletes it

**Code reviews (manual, but self-served):**
- Reviewer writes review doc to `.agent-harness/reviews/review-<timestamp>.md` via `leave-review.md`
- Next session's coder reads it via `pick-up-review.md`, fixes issues, deletes it when complete

Both are gitignored. No emails or external notifications needed — the coder picks them up at session start.

## Concurrent Agents (Worktrees)

Works with git worktrees for parallel agent sessions:

```bash
# Main branch — Opus planning
cd main-branch
# Read: .agent-harness/skills/planner/planner-start.md

# Feature branch — Sonnet coding
git worktree add ../my-feature -b feature/my-feature
cd ../my-feature
# Read: .agent-harness/skills/coder/coder-start.md

# Review branch — Codex reviewing
git worktree add ../review feature/my-feature
cd ../review
# Read: .agent-harness/skills/reviewer/reviewer-start.md
```

Each worktree generates its own `bootstrap.md` (role-specific context). Memory and map are shared across all worktrees. Write contention is handled by file locking.

## Memory System

Two-tier: a thin index always available, full entries loaded on demand.

**Categories:** `architecture` | `pattern` | `decision` | `convention` | `mistake`

```bash
harness memory save --category mistake "Don't use assertEqual for async — use await expect"
harness memory save --category architecture "Auth uses middleware chain, not decorators"
harness memory query "async testing" --brief
```

Memories decay over time (-0.05 relevance/day). Mistakes decay slower (-0.02/day). Accessed entries get a +0.3 bump. Entries below 0.1 relevance are pruned automatically.

## Token Budget

| Component | Characters | ~Tokens |
|---|---|---|
| Bootstrap (all sections) | ~2000 chars | ~500 tokens |
| Memory query (full, 5 entries) | ~2500 chars | ~625 tokens |
| Map drill (one module) | ~1600 chars | ~400 tokens |

Agents spend tokens on reasoning, not context rebuilding.

```
├── memory/              # persistent agent memory
├── rules/               # project conventions
├── reviews/             # transient code review docs (gitignored)
├── handoff/             # transient session handoff docs (gitignored)
├── logs/                # test results
├── scripts/             # all automation
└── templates/           # harness init starters
```

## Running the Tests

The harness ships with a pytest suite covering memory, pruning, permissions injection, bootstrap generation, and CLI integration.

```bash
# Install test dependency (pyyaml is already required)
pip3 install pytest

# Run from inside .agent-harness/
cd .agent-harness
python3 -m pytest tests/ -v
```

Expected output: 60 tests, all passing, ~2s.

Each test gets an isolated temp directory — no real harness data is touched.

## Configuration

Edit `.agent-harness/config.yaml`:

- `project.type` — template type
- `budgets.*` — token/char limits per command
- `memory.*` — max entries, decay rates, prune threshold
- `tests.test_command` — your test runner command
- `roles.*` — permissions and context per role
